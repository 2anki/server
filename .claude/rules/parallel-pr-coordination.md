# Parallel PR coordination

Lessons from shipping 11 PRs in two waves with parallel agents. Read this before kicking off a multi-PR batch (more than two PRs in flight at the same time) or before invoking `/spec-draft-pr` more than once in a row.

| Requirement | Do instead |
| --- | --- |
| Do not run `/spec-draft-pr` (or any `git checkout -b` from a feature branch) without first resetting the working tree to `main`. Specs drafted in sequence stack on each other — each new branch carries the previous branches' spec files as additions, and when the implementation PRs squash-merge they re-introduce sibling specs into `main` as carryover. | `git checkout main && git pull` between every spec-draft. Each spec branch starts from a clean main. If you forgot, expect a `chore: remove implemented specs` cleanup PR after the wave merges. |
| Do not trust `isolation: "worktree"` to survive a session interruption. Agents whose isolated worktrees get cleaned up may silently resume in the main checkout. Multiple agents writing to the same tree overwrite each other and pollute Al's pre-existing local state. | Brief every spawned engineer agent to `pwd` + `git rev-parse --show-toplevel` and **halt before any commit** if the path doesn't match their assigned worktree. The agent should stop and report, not commit and pollute. |
| Do not implement on a stale spec branch. By the time the spec is approved, several other PRs have merged to main; long-lived branches accumulate conflicts on shared files (`changelog.ts`, `App.tsx`, `sitemap.xml`, `FEATURE.md`) and on overlapping feature areas. | `/implement` step 1 is `gh pr checkout` followed by `git fetch origin main && git rebase origin/main`. Step 2 is reading the spec. Step 3 is the plan. Never write code on a branch that hasn't been rebased on origin/main today. |
| Do not let parallel teammates touch the same shared files (`web/src/pages/WhatsNewPage/changelog.ts`, `web/src/App.tsx`, `web/scripts/prerenderLandingPages.ts`, `web/public/sitemap.xml`) without flagging the overlap in their brief. They will conflict at merge time, every time. | When spawning a parallel batch, enumerate which shared files each teammate will touch (one line each in the brief). Coordinate the merge order: smallest-scope PR merges first; subsequent PRs rebase against the new main. Plan for one `chore: remove implemented specs` cleanup PR after each wave. |
| Do not `git add -A` or `git add .` from a polluted working tree. If multiple agents wrote to the main checkout (whether by accident or because their worktrees vanished), an "add all" stages everyone's work as one teammate's commit. | Stage every file by explicit path. Before any `git add` in a multi-agent round, run `git status` and confirm the file belongs to the agent that's about to commit. When in doubt, halt and report file ownership back to the lead. |
| Do not skip the post-wave `Documentation/specs/` audit. Even when each teammate `git rm`s their own spec correctly, sibling spec files leak through stacked-branch carryover, and the directory accumulates done-spec files that look real but are zombies. | After every wave merges, run `git ls-tree -r --name-only origin/main -- Documentation/specs/` and remove any spec whose implementation PR is already merged. One small `chore: remove implemented wave-N specs` PR per wave keeps the directory honest. |

## Wave coordination playbook

For a batch of N PRs from N parallel agents:

1. **Before spawning**: reset to a clean main, identify shared files each PR will touch, write down the merge order (smallest-scope first), and decide which PRs depend on others (those go in wave 2).
2. **During spawn**: every agent gets `mode: "plan"` + `isolation: "worktree"` + a brief that names (a) their assigned file scope, (b) the shared files to flag in their plan, (c) `pwd`-verify before committing.
3. **During implementation**: review every plan against the brief; reject any plan whose file list strays outside scope (extending scope is allowed but must be explicit). If an agent reports they're in the main checkout instead of their worktree, route them to a fresh `/tmp/wt-<name>` and have them `cp` only their own files.
4. **During merge**: rebase + force-push-with-lease + `gh pr merge --auto --squash` for each PR sequentially. Use `Monitor` to wait on each merge instead of polling. Resolve the changelog conflict by stacking the newest entry at the top.
5. **After merge**: audit `Documentation/specs/`, ship the cleanup PR, delete local `rebase/*`, `rescue/*`, `worktree-agent-*`, and merged spec branches. Worktrees first (largest disk recovery), branches second.
